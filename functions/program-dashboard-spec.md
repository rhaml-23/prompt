# Program Dashboard Spec
**Version:** 1.0
**Purpose:** Govern the generation of per-program compliance dashboards and the portfolio briefing page. All dashboards use the 62443 gold-standard light-theme design system (dark navy header, `--bg: #f1f5f9`, CSS variables extracted from `runs/62443/dashboard.html`).
**Governed by:** `config/constitution.md`
**Invokes:** `engine/quality-gate-spec.md` (Gate 1 — constitutional alignment, Gate 2 — structural completeness)

---

## Constitutional Guidance

- **Say the true thing** (Article IV.1) — stale data is flagged with a staleness notice. Missing sections are rendered as `[DATA UNAVAILABLE — source: path]` rather than omitted or left blank. Coverage percentages and risk counts are reported as-is; nothing is inflated or softened.
- **Protect the downstream** (Article IV.2) — dashboards are read by lead program managers, stakeholders, and auditors. Gaps not surfaced here become invisible. Coverage and risk data are shown at face value; no compensating narrative is added to obscure gaps.
- **Surface uncertainty** (Article IV.4) — `[DATA UNAVAILABLE]` is the correct response to a missing section. A staleness notice is shown when the run date is past the recommended next-run date. `[INFERRED]` is preserved where the source data uses it.
- **Good enough calibration** (Article IV.14) — these are operational artifacts. Accurate and clear is the goal. Formatting serves legibility, not aesthetics.

## Renderers

Two scripts implement this spec. They share the same CSS design system.

| Script | Input | Output | Purpose |
|---|---|---|---|
| `scripts/program_dashboard_renderer.py` | `runs/[PROGRAM]/latest.json` | `public/[PROGRAM]/index.html` | Per-program light-theme dashboard (gold standard) |
| `scripts/portfolio_renderer.py` | `data/portfolio/latest.json` | `ui/portfolio.html` | Portfolio briefing — all programs at a glance |
| `scripts/dashboard.py` | `runs/*/latest.json` | `public/programs.html` | Combined multi-program dark-theme view (legacy, unchanged) |

`dashboard.py --program [slug]` (dark-theme single-program flag) is available as a CLI tool but is **not called by `scripts/build_pages.py`**. Per-program pages in `public/[slug]/index.html` are exclusively generated by `program_dashboard_renderer.py`.

| `--bg` | `#f1f5f9` | Page background |

| `--surface-2` | `#f8fafc` | Card header, secondary surfaces |

| `--high` / `--high-light` | `#ea580c` / `#ffedd5` | High risk, warnings |

No external fonts, no JavaScript, no CDN dependencies. Renders correctly when printed to PDF.

## Schema Normalisation

Both renderers support schema 1.1 (legacy) and schema 2.0 (standard) run JSON transparently. See `functions/control-coverage-spec.md → Schema Variants` for the full mapping. The `_normalize_coverage()` helper in both scripts handles both shapes.

## Processing Passes — `program_dashboard_renderer.py`

### Pass 1 — Data Load

Read `runs/[PROGRAM]/latest.json`. Extract using schema-agnostic helpers:
- `program_name` (schema 2.0 top-level) or `run_manifest.program_name` (schema 1.1)
- `overall_health` (schema 2.0) or `program_state.overall_health` (schema 1.1)
- `run_date` (schema 2.0) or `run_manifest.run_date` (schema 1.1)
- `phase`, `scope.frameworks` or `intake_output.scope.frameworks`
- `people.roster` or `intake_output.people.roster`
- `monitoring_output.decision_queue`, `.watch_list`, `.escalation_items` (schema 1.1) or `commitments` (schema 2.0)
- `flags`, `risk_register`, `vendor_output`, `calendar_events`, `next_run_recommendation`

If `latest.json` does not exist, exit with a clear error message — do not fabricate data.

### Pass 2 — Coverage Normalisation

Call `_normalize_coverage(run)` to produce a unified `{families, totals, framework, assessment_date}` shape regardless of source schema. See the normaliser logic in the script and the schema table in `functions/control-coverage-spec.md`.

If no coverage data is present in either `control_coverage` or `control_coverage_matrix`, render the coverage section as `[DATA UNAVAILABLE]` with source path.

### Pass 3 — Section Assembly

Render sections in order. Each section degrades gracefully:

| Section | Source field | Missing data behaviour |
|---|---|---|
| Header + health badge | `program_name`, `overall_health`, `run_date` | Always present — use `Unknown` as fallback |
| Staleness notice | `next_run_recommendation.suggested_date` vs today | Omit if date not present or not overdue |
| Status bar | `one_line_status` or `program_state.one_line_status` | Omit if absent |
| Escalations | `monitoring_output.escalation_items` | Omit if absent |
| KPI strip | `risk_register.open`, `flags`, `monitoring_output.decision_queue` | Omit individual cards with zero values |
| Control coverage | `_normalize_coverage()` | `[DATA UNAVAILABLE]` with path |
| Risk register | `risk_register.items`, `.open` | `[DATA UNAVAILABLE]` with path |
| Decision queue | `monitoring_output.decision_queue` or `commitments` | Omit if empty |
| Watch list | `monitoring_output.watch_list` | Omit if absent |
| Evidence calendar | `calendar_events` | Omit if absent |
| Vendor scorecard | `vendor_output.scorecard` | Omit if absent or not a dict |
| Flags summary | `flags` | Omit if absent |
| Next run recommendation | `next_run_recommendation` | Omit if absent |

### Pass 4 — HTML Render

Assemble the full HTML document. Inline all CSS. No external resources. Output to `--output` path (default `runs/[slug]/dashboard.html`; `build_pages.py` sets it to `public/[slug]/index.html`).

### Pass 5 — Provenance Log Write

After successful write, log to provenance:
```bash
python scripts/provenance_log.py write \
  --spec "functions/program-dashboard-spec.md" \
  --output "public/[PROGRAM]/index.html" \
  --output-type dashboard_html \
  --program "[PROGRAM]" \
  --purpose "Per-program light-theme dashboard — [health] health" \
  --reusability artifact \
  --quality-gate pass
```

## Processing Passes — `portfolio_renderer.py`

### Pass 1 — Data Load

Read `data/portfolio/latest.json`. Key fields used:
- `portfolio_health.{total_programs, red, yellow, green, total_open_decisions, total_blockers, total_escalations}`
- `programs[].{program_slug, health, health_reason, one_line_status, phase, frameworks, open_decision_count, blocker_count, escalation_count, owner_gap_count, vendors, last_run_date, next_run_recommended, next_run_intent}`
- `cross_program_signals.{deadline_clustering, resource_contention, intel_overlap}`
- `suggested_actions`

If `data/portfolio/latest.json` does not exist, exit with error — do not fabricate portfolio state.

### Pass 2 — Link Resolution

For each program slug, attempt to resolve:
- Dashboard link: `public/[slug]/index.html` (generated) or `runs/[slug]/dashboard.html` (committed fallback)
- Auditor link: `ui/[slug]-auditor-[today].html` or most recent file matching `ui/[slug]-auditor-*.html`

Links that cannot be resolved are rendered as disabled ("not generated") rather than pointing to dead URLs.

### Pass 3 — Section Assembly

Sections rendered in order:
1. Header with portfolio health and program counts
2. KPI strip (green / yellow / red / decisions / blockers / escalations)
3. Per-program cards (one per entry in `programs[]`)
4. Cross-program signals (deadline clustering, resource contention)
5. Suggested actions

### Pass 4 — HTML Render and Provenance

Assemble and write HTML, then log provenance.

## Build Pipeline Integration

`scripts/build_pages.py` calls both renderers as part of its `regenerate_dashboards` function:

```
Step 1 — dashboard.py             → public/programs.html       (combined dark view)
Step 2 — fleet_dashboard.py       → public/fleet.html
Step 3 — program_dashboard_renderer.py [slug]  → public/[slug]/index.html  (per program)
Step 4 — auditor_view_renderer.py [slug]       → ui/[slug]-auditor-[today].html
Step 5 — portfolio_renderer.py                 → ui/portfolio.html
```

The committed `runs/*/dashboard.html` files (hand-crafted gold-standard HTML) are preserved as fallback via `copy_program_dashboards`. Step 3 generates fresh output on every CI push, overriding the fallback only when successful.

## Quality Gate

Before this spec is considered complete for any run:

**Gate 1 — Constitutional Alignment**
- [ ] No section contains fabricated data
- [ ] Missing sections are rendered as `[DATA UNAVAILABLE]` not blank
- [ ] Stale data carries a staleness notice
- [ ] `[INFERRED]` is preserved where source data uses it

**Gate 2 — Structural Completeness**
- [ ] HTML is valid and renderable without JavaScript or external assets
- [ ] All required sections are present or explicitly marked unavailable
- [ ] Output path was written successfully (no silent failures)
- [ ] Provenance was logged on completion
